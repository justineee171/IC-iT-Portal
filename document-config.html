Document Server Setup

Objective

Properly document your server setup to ensure maintainability, troubleshooting efficiency, and team collaboration. Clear documentation helps reduce downtime, improves compliance, and serves as a reference for future upgrades or audits.

Step 1: Record Server Details

  • Server name, IP address, MAC address, and role (e.g., DNS, File Server).
  • Operating system version, build number, and update status.
  • Installed applications and their versions.
  • Serial numbers and hardware specs (CPU, RAM, disk, NICs).

Step 2: Capture Network Configuration

  • IP address assignment (static or dynamic).
  • Subnet mask, default gateway, and DNS servers.
  • VLAN or switch port mapping if applicable.
  • Firewall rules and port configurations specific to the server.

Step 3: Document Services and Roles

  • List installed roles (e.g., Active Directory, DHCP, IIS).
  • Configuration paths, service dependencies, and key parameters.
  • Startup type (automatic/manual) and current service status.
  • Ports and protocols used by each role or application.

Step 4: Security Configurations

  • User accounts and permission groups with access levels.
  • Audit policies and logging settings.
  • Antivirus software and scanning schedule.
  • Remote access settings (RDP/SSH) and restrictions.

Step 5: Backup and Recovery

  • Backup software used and backup schedule.
  • Location of backup files (local, remote, or cloud).
  • Retention policy and last successful restore test.
  • Disaster recovery plan or checklist.

Step 6: Maintain Change Logs

  • Log all changes: updates, configuration tweaks, and installed packages.
  • Include timestamps, responsible personnel, and reasons for changes.
  • Store logs securely and ensure they are version controlled if possible.

Step 7: Use Standardized Templates

  • Use consistent templates for all server documentation.
  • Templates may include sections for hardware, software, network, security, users, and maintenance.
  • Include diagrams such as network topology or server rack layout if helpful.

Best Practices

  • Store documentation in a shared, secure location (e.g., internal wiki or versioned repository).
  • Keep documentation updated after every major change.
  • Limit access to sensitive configuration files but allow read access for relevant team members.
  • Review documentation regularly as part of maintenance checks.
